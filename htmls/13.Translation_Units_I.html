Basic Concepts
Translation Unit
Header File and Source File
Header les allow dening interfaces (.h, .hpp, .hxx), while keeping the
implementation in separated source les (.c, .cpp, .cxx).
Translation Unit
A translation unit (or compilation unit) is the basic unit of compilation in C++. It
consists of the content of a single source le, plus the content of any header le
directly or indirectly included by it
A single translation unit can be compiled into an object le, library, or executable
program. The nal program or library is the combination of all translation units

Local and Global Scope
Scope
The scope of a variable/function/object is the region of the code within the entity
can be accessed
Local Scope / Block Scope
Entities that are declared inside a function or a block are called local variables.
Their memory address is not valid outside their scope
Global Scope / File Scope / Namespace Scope
Entities that are dened outside all functions.
They hold a single memory location throughout the life-time of the program

Linkage
Symbols
Symbols are named items (identiers) in a program and they refer to entities such as
variables, functions, classes
Linkage
Linkage refers to the visibility of symbols to the linker
There are three ways to link symbols:
No linkage
Internal linkage
External linkage
10/59
Linkage
No Linkage
No linkage refers to symbols in the local scope of declaration and not visible to the
linker.
Internal Linkage
Internal linkage refers to symbols accessible only in scope of a single translation
unit. The same symbol name has a dierent memory address in distinct translation
units.
External Linkage
External linkage refers to symbols that are accessible outside a single translation
unit. Symbols with external linkage have an unique memory address through the
whole program. The same name in dierent translation units causes conicts.
11/59
Visibility 1/2
Visibility
Symbol visibility determines whether a symbol is exported from a shared library or
executable, making it available for dynamic linking.
The symbol table holds information about these symbols, including their names,
addresses, linkage, and visibility.
Visibility attributes:
default The symbol is visible to external binaries and can be linked against.
hidden The symbol is not visible to external binaries; it is only accessible within the
dening component.
Multiple libraries that export symbols with the same name could potentially cause
undened behavior at runtime.
12/59
Visibility 2/2
Symbol visibility and linkage are related but distinct concepts.
Symbols with internal linkage cannot be exported (hidden visibility).
Symbols with external linkage are exported if they have default visibility.
GCC/Clang Default visibility: default . The compilers allow the opposite behavior
with -fvisibility=hidden .
MSVC Default visibility: hidden .
Additionally, it is possible to control individual symbol visibility:
__attribute__((visibility("hidden")))
void internal_function(); // GCC/Clang
__attribute__((visibility("default")))
int exported_variable = 3;
__declspec(dllexport) // MSVC
void exported_function();
__declspec(dllimport)
void imported_function();

Storage Duration
The storage duration (or duration class) determines the duration of a variable,
namely when it is created and destroyed
Storage Duration Allocation Deallocation
Automatic Code blo ck start Code blo ck end
Static Program start Program end
Dynamic Memory allocation Memory deallocation

en.cppreference.com/w/cpp/language/storage_duration
14/59
Storage Duration 2/2
Automatic storage duration. Local variables temporary allocated on registers or
stack (depending on compiler, architecture, etc.).
If not explicitly initialized, their value is undened
Static storage duration. The storage of an object is allocated when the program
begins and deallocated when the program ends.
If not explicitly initialized, it is zero-initialized
Dynamic storage duration. The object is allocated and deallocated by using
dynamic memory allocation functions ( new/delete ).
If not explicitly initialized, its memory content is undened
Thread storage duration C++11. The object is allocated when the thread
begins and deallocated when the thread ends. Each thread has its own instance of
the object

Storage Duration Examples

int v2; // automatic duration
auto v3 = 3; // automatic duration
auto array = new int[10]; // dynamic duration (allocation)
} // array, v2, v3 variables deallocation (from stack)
// the memory associated to "array" is not deallocated
int main() {
f();
}
// main end: v1 is deallocated

18/59
static Keyword for Local Variables
static local variables are allocated when the program begins, initialized when the
function is called the rst time, and deallocated when the program ends
int f() {
static int val = 1;
val++;
return val;
}
int main() {
cout << f(); // print 2 ("val" is initialized)
cout << f(); // print 3
cout << f(); // print 4
}
19/59
static Keyword for Global Variables
static global variables or functions are visible only within the translation unit where
they are declared internal linkage
Non- static global variables or functions with the same name in dierent translation
units produce name collision (or name conict) multiple denitions at link-time
int var1 = 3; // external linkage
// (in conflict with variables in other
// translation units with the same name)
static int var2 = 4; // internal linkage (visible only in the
// current translation unit)
void f1() {} // external linkage (could conflict)
static void f2() {} // internal linkage
20/59
Anonymous Namespace 1/2
A namespace with no identier is called unnamed/anonymous namespace
Entities within an anonymous namespace have internal linkage and, therefore, are used
for declaring unique identiers, visible only in the same source le
Anonymous namespace vs. global static functions/variables:
Entities withing an anonymous namespace have the same properties of static
declarations at global scope
In addition, anonymous namespaces allow type declarations and class denitions
Anonymous namespaces are less verbose than static variables/functions but,
entities within an anonymous namespace are less visible if the scope contains
many lines
21/59
Anonymous Namespace 2/2
main.cpp
# include <iostream>
namespace { // anonymous, internal linkage
void f() { std::cout << "main"; }
using my_int = int; // not possible
// with 'static'
} // namespace
int main() {
f(); // print "main"
}
source.cpp
# include <iostream>
namespace { // anonymous, internal linkage
void f() { std::cout << "source"; }
using my_int = unsigned; // no conflicts
} // namespace
int g() {
f(); // print "source", no conflicts
}
22/59
extern Keyword
extern keyword is used to declare the existence of global variables or functions in
another translation unit external linkage
the variable or function must be dened in one and only one translation unit
it is redundant for functions
it is necessary for variables to prevent the compiler to associate a memory location
in the current translation unit
Note: if the same identier within a translation unit appears with both internal and external
linkage, the behavior is undened

Static Initialization Order Fiasco 1/2
In C++, the order in which global variables are initialized at runtime is not dened.
This introduces a subtle problem called static initialization order asco
source.cpp
int f() { return 3; } // run-time function
int x = f(); // run-time evalutation
main.cpp
extern int x;
int y = x; // run-time initialized
int main() {
cout << y; // print "3" or "0" depending on the linking order
}
26/59
Static Initialization Order Fiasco 2/2
source.cpp
constexpr int f() { return 3; } // compile-time/run-time function
constinit int x = f(); // compile-time initialized (C++20)
main.cpp
constinit extern int x; // compile-time initialized (C++20)
int y = x; // run-time initialized
int main() {
cout << y; // print "3"!!
}

No Linkage: Local variables, functions, classes

Internal Linkage:
(not accessible by other translation units, no conicts, dierent memory addresses)

Global Variables:

One Denition Rule
(ODR)
One Denition Rule (ODR)
(1) In any (single) translation unit, a template, type, function, or object, cannot
have more than one denition
- Compiler error otherwise
- Any number of declarations are allowed
(2) In the entire program, an object or non-inline function cannot have more
than one denition
- Multiple denitions linking error otherwise
- Entities with internal linkage in dierent translation units are allowed, even if their
names and types are the same
(3) A template, type, or inline functions/variables, can be dened in more than
one translation unit. For a given entity, each denition must be the same
- Undened behavior otherwise
- Common case: same header included in multiple translation units
34/59
ODR - Point (1), (2)
header.hpp:
void f(); // DECLARATION
main.cpp:
# include "header.hpp"
# include <iostream>
int a = 1; // external linkage
// int a = 7; // compiler error, Point (1)
extern int b;
static int c = 2; // internal linkage
int main() {
std::cout << a; // print 1
std::cout << b; // print 5
std::cout << c; // print 2
f();
}
source.cpp:
# include "header.hpp"
# include <iostream>
// linking error, multiple definitions
// int a = 2; // Point (2)
int b = 5; // ok
// internal linkage
static int c = 4; // ok
void f() { // DEFINITION
// std::cout << a; // 'a' is not visible
std::cout << b; // print 5
std::cout << c; // print 4
}

37/59
ODR - Point (3)
ODR Point (3): A template, type, or inline functions/variables, can be
dened in more than one translation unit
The linker removes all denitions of an inline / template entity except one
All denitions must be identical to avoid undened behavior due to arbitrary
linking order
inline / template entities have a unique memory address across all translation
units
inline / template entities have the same linkage as the corresponding
variables/functions without the specier
38/59
inline Functions/Variables 1/2
inline
inline specier allows a function or a variable (in C++17) to be identically
dened (not only declared) in multiple translation units
inline is one of the most misunderstood features of C++
inline is a hint for the linker. Without it, the linker can emit multiple
definitions error
inline entities cannot be exported, namely, used by other translation units even
if they have external linkage (related warning: -Wundefined-inline )
inline doesnt mean that the compiler is forced to perform function inlining. It
just increases the optimization heuristic threshold
39/59
inline Functions/Variables 2/2
void f() {}
inline void g() {}
f() :
Cannot be dened in a header included in multiple source les
The linker issues a multiple denitions error
g() :
Can be dened in a header and included in multiple source les
40/59
constexpr and inline
constexpr functions are implicitly inline
constexpr variables are not implicitly inline . C++17 added inline variables
void f1() {} // external linkage
// potential multiple definitions error
constexpr void f2() {} // external linkage, implicitly inline
// multiple definitions allowed
constexpr int x = 3; // internal linkage
// different files allows distinct definitions
// -> different addresses, code bloat
inline constexpr int y = 3; // external linkage unique memory address
// -> potential undefined behavior

Summary
Undened Behavior - inline Function
main.cpp:
# include <iostream>
inline int f() { return 3; }
void g();
int main() {
std::cout << f(); // print 3
std::cout << g(); // print 3!!
} // not 5
source.cpp:
// same signature and inline
inline int f() { return 5; }
int g() { return f(); }
The linker can arbitrary choose one of the two denitions of f() . With -O3 , the
compiler could inline f() in g() , so now g() return 5
This issue is easy to detect in trivial examples but hard to nd in large codebase
Solution: static or anonymous namespace
